Modular Dashboard Refactor

Ghostfolio — Single-Canvas, Drag-and-Drop Composition

Replace the route-driven shell with a registry-backed grid where every existing feature becomes a placeable, resizable, persistable module.

Executive Action Plan · Blitzy · AAP § 0.6.1.9

01

Why

From Routes to Composition

A persistent, user-shaped workspace replaces twenty-two lazy-loaded screens.

Why a Modular Dashboard

Pain Today

  • Twenty-two lazy routes plus tab shells fragment the user journey.
  • AI chat lives buried beneath the portfolio tab navigation.
  • Every user sees the same layout, regardless of role or focus.
  • Header and footer chrome consume vertical space without payoff.
  • No mechanism exists to remember a user’s preferred view.

Outcome After Refactor

  • One root route, one canvas, infinite user-shaped layouts.
  • AI chat is a co-equal grid module, on by default for new users.
  • Per-user layouts persist server-side keyed on JWT-derived identity.
  • Full viewport for content; chrome reduces to <router-outlet />.
  • First-visit auto-opens the catalog; nothing to configure to start.

The refactor is additive to the data layer: every existing service (Portfolio, Symbol, AI Chat, Rebalancing, FinancialProfile) is consumed unchanged. Module wrappers are thin shells around presentation components that already render today.

API Contract

Risk & Validation

What We Watch and How We Verify

Anticipated failure modes, the controls that contain them, and the gates that prove the work.

Risk Register and Mitigations

Risk Severity Mitigation
Cross-user layout read or writeLowuserId always sourced from request.user.id (Engineering Rule 5); Prisma upsert.where.userId never echoes the request body.
Unauthenticated layout accessLowAuthGuard('jwt') rejects pre-permission with 401; HasPermissionGuard denies DEMO and INACTIVE roles with 403.
Malicious payload via PATCHLowclass-validator DTO bounds items to fifty entries, integer-bounds on cols/rows/x/y, and moduleId length cap.
Migration conflict with existing schemaLowSchema located at prisma/schema.prisma (verified); model appended after FinancialProfile; Prisma 7.x explicit back-relation per Decision D-013.
Drag/resize jank under zone-based CDMediumPer-frame layout math runs NgZone.runOutsideAngular; canvas is OnPush; re-enters zone at drag-stop boundary; 100 ms SLO measured per AAP § 0.6.3.3.
Save storms under rapid drag/resizeMediumrxjs debounceTime(500) in LayoutPersistenceService collapses bursts into a single PATCH; module wrappers MUST NOT call save APIs (Rule 4).
Module ad-hoc insertion bypassing registryLowCanvas resolves components only via ModuleRegistryService.getByName(...) (Rule 3); static grep verifies no hard-coded component imports inside the canvas.
Orphaned layout rows on user deletionLowonDelete: Cascade on the userId FK mirrors the FinancialProfile precedent at schema.prisma:374.

Outcome

Ship It

A modular workspace your users will shape, with auditable provenance for every change.

Get Started in Five Steps

  1. Read the Agent Action Plan. Sections 0.1 through 0.9 capture intent, scope, integration map, design-system compliance, and execution plan.
  2. Install the grid engine. npm install angular-gridster2@21.0.1. The package ships its CSS via ViewEncapsulation.None in fesm2022/angular-gridster2.mjs; no @import from styles.scss is required (Decision D-024).
  3. Run the migration. npx prisma migrate dev --name add-user-dashboard-layout creates the table, FK with cascade, and regenerates the typed client.
  4. Verify the build. npx nx build client && npx nx build api must complete with zero errors and zero warnings.
  5. Walk the gates. Eleven acceptance gates in AAP § 0.8.6; ten engineering rules in § 0.8.1; segmented PR review per CODE_REVIEW.md.
