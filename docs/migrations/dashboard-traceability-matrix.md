# Dashboard Refactor Traceability Matrix

This document is the canonical bidirectional traceability matrix for the **Modular Dashboard Refactor** of Ghostfolio v3.0.0 — the work item scoped by AAP § 0.1.1 (Core Feature Objective). Its authority is AAP § 0.8.2.2 (Explainability project rule), which mandates that every removed, preserved, and added construct in the refactor be mapped to its target file with **100% coverage, zero gaps**. The two siblings of this artifact are the decision log at `../decisions/agent-action-plan-decisions.md` (which records the per-decision rationale, alternatives considered, and risk trade-offs for `D-024` and `D-025` from the Modular Dashboard Refactor Checkpoint 1 Code Review remediation pass and `D-026` through `D-035` from the modular dashboard refactor design-decisions documentation pass) and the observability runbook at `../observability/dashboard-layout.md` (which documents the metrics, structured logs, dashboards, and local verification procedures for the new layout endpoints). The **forward** table (_requirement → file_) answers "where is this requirement implemented?" and is the primary tool for change-impact analysis: given a requirement, locate every file that must be touched to ship or revert it. The **reverse** table (_file → requirement_) answers "what does this file address?" and is the primary tool for refactor-and-review of a single file: given a path, locate every requirement it carries. The complementary `## Removed Surfaces`, `## Preserved Surfaces`, and `## Deviation Log` sections together partition every file change enumerated in AAP § 0.7.1 into exactly one of four categories — created, modified, removed, or preserved — with no overlaps.

## Deviation Log

### Intentional Deviation: ChatPanelComponent → Standalone Grid Module

The user prompt explicitly mandates a single deviation from the existing tech spec § 7.4.1.2 ("the component is rendered below the `mat-tab-nav-bar` of the portfolio page and persists across all six portfolio tabs"). Quoted verbatim from the prompt:

> "This refactor explicitly deviates from the tech spec's definition of `ChatPanelComponent` as embedded within `portfolio-page.html:32`. The chat panel MUST be treated as a standalone grid module co-equal with all other modules. This deviation is intentional."

| Aspect             | Detail                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                              |
| ------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Source mount-point | `apps/client/src/app/pages/portfolio/portfolio-page.html` line 32 (`<app-chat-panel></app-chat-panel>`)                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                             |
| Target wrapper     | `apps/client/src/app/dashboard/modules/chat/chat-module.component.ts`                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                               |
| Decision log entry | `D-026` — recorded in `../decisions/agent-action-plan-decisions.md`                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                 |
| Rationale          | The canvas-everywhere mandate from AAP § 0.1.1 requires every user-facing feature to be a self-contained, independently placeable grid module. AI chat is a co-equal feature with portfolio overview, holdings, transactions, and analysis; embedding it as a fixed sidebar would contradict the modular composition primitive. The deviation preserves the existing `ChatPanelComponent` internal implementation entirely (selector `app-chat-panel`, the constants `MAX_CLIENT_TRANSMITTED_MESSAGES = 5`, `COLLAPSED_WIDTH_REM = '2.75rem'`, `DEFAULT_EXPANDED_WIDTH_PX = 280`, `MIN_EXPANDED_WIDTH_PX = 200`, `MAX_EXPANDED_WIDTH_PX = 600`, `AiChatService` consumption, SSE handling) — only the mount-point host is replaced. |
| Trade-off          | The chat panel is no longer always-visible across tabs; users must explicitly add it to their dashboard. This is mitigated by (a) the catalog auto-opening on first visit (Rule 10), (b) the chat module appearing as a co-equal entry in the module catalog, and (c) the user's prompt explicitly identifying this as the desired UX direction.                                                                                                                                                                                                                                                                                                                                                                                    |
| Cross-references   | AAP § 0.1.1 (Core Feature Objective — chat-as-co-equal-module mandate); AAP § 0.4.1.7 (Removed Integration Surfaces — `portfolio-page.html:32` listed as the source); AAP § 0.7.2 (Intentional Deviation From Tech Spec — verbatim record); `D-026` in `../decisions/agent-action-plan-decisions.md` (decision rationale and alternatives).                                                                                                                                                                                                                                                                                                                                                                                         |

The chat panel's internal implementation is **NOT modified** by this refactor. The wrapper at `apps/client/src/app/dashboard/modules/chat/chat-module.component.ts` is a thin shell that imports `ChatPanelComponent` via its existing public selector `app-chat-panel`, places it inside the canonical `GfModuleWrapperComponent` chrome (header, drag handle, kebab, remove), and registers itself with `ModuleRegistryService` under the stable name `chat`.

This matrix is maintained alongside the dashboard refactor; any future material change — a new module, a renamed file, a reversed decision, or a new interface — MUST be reflected in both the forward and reverse tables and, where applicable, in the Removed Surfaces, Preserved Surfaces, or Deviation Log sections. Partial updates that leave the four tables and the deviation log out of sync compromise the traceability guarantee that AAP § 0.8.2.2 (Explainability) mandates and undermine the audit-grade utility of this document for downstream reviewers.

When a file is renamed or split, replace the path in-place in both tables (renames) or split the row into multiple rows (splits) so that every shipped path resolves to exactly one row in each direction. When a new deviation is added, append a new H3 subsection under `## Deviation Log` using the same six-aspect template (Source mount-point / Target wrapper / Decision log entry / Rationale / Trade-off / Cross-references) that the chat-panel deviation establishes. When a preservation anchor is rewritten or relocated, update the corresponding row of `## Preserved Surfaces` to point at the new location and cite the reviewing decision-log entry that authorised the change.

The matrix is intentionally append-only at the row level — a superseded row should be marked `Superseded by <new row>` rather than deleted, preserving the historical audit trail.
